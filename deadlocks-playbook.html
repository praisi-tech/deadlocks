Deadlock Playbook

Materi Lengkap, Tracing Matriks, Implementasi C++, & Latihan Soal (Multiple Resources)

Definisi Deadlock

"A set of processes is deadlocked if each process in the set is waiting for an event that only another process in the set can cause."
— Biasanya disebabkan oleh proses yang menunggu pelepasan resource yang sedang ditahan oleh proses lain.

4 Kondisi Penyebab Deadlock (Four Conditions)

1. Mutual Exclusion

Setiap resource hanya diberikan pada tepat 1 proses pada satu waktu (atau tersedia bebas).

2. Hold and Wait

Proses yang sedang menahan resource diperbolehkan meminta resource tambahan yang baru.

3. No Preemption

Resource yang sudah diberikan tidak bisa diambil secara paksa oleh sistem. Harus dilepas mandiri.

4. Circular Wait

Terdapat rantai melingkar dari 2 proses atau lebih di mana masing-masing menunggu resource dari member berikutnya.

Strategi Penanganan (Strategies for Deadlocks)

1. The Ostrich Algorithm

Berpura-pura tidak ada masalah (mengabaikan deadlock). Masuk akal jika deadlock sangat jarang terjadi dan biaya pencegahannya mahal (Pendekatan UNIX & Windows).

2. Detection and Recovery

Membiarkan deadlock terjadi, lalu mendeteksinya dan melakukan recovery. (Fokus pada algoritma di tab Tracing).

3. Dynamic Avoidance

Alokasi resource yang berhati-hati menggunakan algoritma seperti Banker's Algorithm (Safe vs Unsafe states).

4. Prevention

Menyerang atau mematahkan salah satu dari 4 kondisi deadlock di atas secara struktural.

Recovery dari Deadlock

  • Preemption: Mengambil paksa resource dari proses lain.
  • Rollback: Menggunakan checkpointing secara berkala dan mengulang proses jika deadlock.
  • Killing Processes: Cara paling sederhana dan kasar; membunuh salah satu proses dalam siklus deadlock.

Implementasi Simulator C++

Source code utuh (deadlock_simulation.cpp) yang mensimulasikan hasil tracing di atas menggunakan struktur data std::vector C++ modern.

#include <iostream>
#include <vector>
#include <iomanip>

using namespace std;

// Number of processes and resources
const int P = 4; // Processes
const int R_num = 5; // Resources

// Function to print a vector
void printVector(const string& name, const vector<int>& vec) {
    cout << name << ": [ ";
    for (int v : vec) { cout << v << " "; }
    cout << "]\n";
}

// Function to check if Request <= Available
bool canBeGranted(const vector<int>& request, const vector<int>& available) {
    for (int i = 0; i < R_num; ++i) {
        if (request[i] > available[i]) return false;
    }
    return true;
}

int main() {
    // 1. Data structure for the note itself (declaring the note)
    
    // Existing resource vector (E)
    vector<int> E = {2, 4, 1, 4, 4};
    
    // Current allocation matrix (C)
    vector<vector<int>> C = {
        {0, 1, 1, 1, 2},
        {0, 1, 0, 1, 0},
        {0, 0, 0, 0, 1},
        {2, 1, 0, 0, 0}
    };
    
    // Request matrix (R)
    vector<vector<int>> R = {
        {1, 1, 0, 2, 1},
        {0, 1, 0, 2, 1},
        {0, 2, 0, 3, 1},
        {0, 2, 1, 1, 0}
    };
    
    // Calculate Available resources (A)
    vector<int> A(R_num, 0);
    vector<int> sum_C(R_num, 0);
    
    for (int j = 0; j < R_num; ++j) {
        for (int i = 0; i < P; ++i) { sum_C[j] += C[i][j]; }
        A[j] = E[j] - sum_C[j];
    }
    
    // Tracing...
    vector<bool> finish(P, false);
    bool process_executed = true;
    
    while (process_executed) {
        process_executed = false;
        for (int i = 0; i < P; ++i) {
            if (!finish[i]) {
                if (canBeGranted(R[i], A)) {
                    for (int j = 0; j < R_num; ++j) { A[j] += C[i][j]; }
                    finish[i] = true;
                    process_executed = true;
                }
            }
        }
    }
    
    // Result check
    bool is_deadlock = false;
    for (int i = 0; i < P; ++i) {
        if (!finish[i]) { is_deadlock = true; }
    }
    
    if (is_deadlock) cout << "\nStatus: Deadlock happened!\n";
    else cout << "\nStatus: System is safe. No deadlock.\n";
    
    return 0;
}

Kumpulan Soal & Pembahasan Tambahan

Berikut adalah beberapa contoh soal latihan seputar Deadlock dan Solusi Banker's Algorithm beserta pembahasannya.

Q1 Terdapat 3 buah proses (P1, P2, P3) dan 3 jenis resource (A, B, C). Total resource yang ada di sistem E = [10, 5, 7]. Jika diketahui Current Allocation dan Maximum Need, apakah sistem dalam keadaan aman (Safe State)?

Current Allocation:
P1: [0, 1, 0]
P2: [2, 0, 0]
P3: [3, 0, 2]

Max Need:
P1: [7, 5, 3]
P2: [3, 2, 2]
P3: [9, 0, 2]

Pembahasan (Banker's Algorithm):

  • Step 1 (Hitung Available): A = E - Sum(Alloc) = [10, 5, 7] - [5, 1, 2] = [5, 4, 5].
  • Step 2 (Hitung Matrix "Need" = Max - Alloc):
    P1: [7, 4, 3]
    P2: [1, 2, 2]
    P3: [6, 0, 0]
  • Step 3 (Tracing Need <= Available):
    - Cek P1: [7, 4, 3] <= [5, 4, 5]? Salah (karena 7 > 5).
    - Cek P2: [1, 2, 2] <= [5, 4, 5]? Benar! P2 dieksekusi, A baru = [5, 4, 5] + [2, 0, 0] = [7, 4, 5].
    - Cek P3: [6, 0, 0] <= [7, 4, 5]? Benar! P3 dieksekusi, A baru = [7, 4, 5] + [3, 0, 2] = [10, 4, 7].
    - Cek P1 lagi: [7, 4, 3] <= [10, 4, 7]? Benar! P1 dieksekusi.

Jawaban: Ya, sistem berada dalam Safe State. Safe Sequence yang ditemukan adalah P2 -> P3 -> P1.

Q2 Menurut materi yang ada (PPT), bagaimana cara kita mencegah (Prevention) sebuah kondisi "Hold and Wait" agar Deadlock tidak terjadi?

Pembahasan:

Kondisi Hold and Wait terjadi ketika proses menahan beberapa resource dan di saat bersamaan ia sedang meminta (request) resource baru. Untuk mencegahnya (seperti yang tercantum pada PPT Slide 23), terdapat dua cara pendekatan utama:

  1. Require processes to request resources before starting: Meminta proses untuk "memesan" atau meminta (request) semua resource yang dibutuhkannya dari awal, sebelum eksekusinya dimulai. Jika semuanya tersedia maka langsung diambil semua. Kelemahannya: sering kali proses tidak tahu persis semua resource yang dibutuhkan di awal.
  2. Variation (Give up all resources): Jika proses butuh resource tambahan baru, proses tersebut dipaksa untuk melepaskan (give up) semua resource yang sedang ditahannya saat ini, lalu me-request ulang semua resource yang ia butuhkan secara sekaligus.
Q3 Jelaskan apa kelemahan dari Ostrich Algorithm dan kapan algoritma ini paling sering digunakan?

Pembahasan:

The Ostrich Algorithm pada dasarnya adalah strategi sistem operasi yang mengabaikan kemungkinan terjadinya deadlock sepenuhnya (seperti burung unta yang menyembunyikan kepalanya di pasir saat ada bahaya).

Kelemahan: Jika sistem terlanjur masuk kondisi deadlock, tidak ada mekanisme internal yang akan meresolvusinya. Seringkali pengguna harus me-restart aplikasi atau mematikan proses secara paksa dari Task Manager.

Kapan Digunakan: Strategi ini digunakan jika probabilitas atau frekuensi terjadinya deadlock sangatlah rendah/jarang (very rarely), dan biaya untuk melakukan deteksi/pencegahan deadlock pada setiap eksekusi program sangatlah mahal (mengurangi performa PC secara drastis). Sistem Operasi modern seperti UNIX dan Windows memakai pendekatan ini demi memprioritaskan performa convenience ketimbang ketelitian ekstrim correctness.
